# 三省六部项目

## 1. 概述

本项目模拟中国古代最著名的政治制度——三省六部制，这是一个高度系统化的政府组织架构，体现了古代中国在国家治理方面的卓越智慧。项目基于OpenClaw构建，是一个分布式AI代理协作框架，模仿中国古代官僚体系，实现任务的有序分配、执行和监控。

### 1.1 项目背景
三省六部制是中国古代封建社会一套中央官制，确立于隋朝，完善于唐朝，此后一直到清末六部制被推翻为止。这套制度将国家政务分为决策、审议、执行三个环节，形成了相互配合又相互制约的权力运行机制。

### 1.2 项目目标
- 实现任务的有序分配与执行
- 建立权责明确的代理协作体系
- 提供标准化的工作流程
- 确保系统的稳定性和可扩展性

## 2. 系统架构

### 2.1 三省架构

#### 中书省（决策机构）
- **职能**：负责政策制定和决策，对应现代企业中的战略规划部门
- **长官**：中书令（正二品），副职为中书侍郎
- **主要职责**：秉承君主旨意，草拟诏令，参与国家重大决策的制定

#### 门下省（审议机构）
- **职能**：负责审核中书省起草的诏令，承担质量控制和审核职能
- **长官**：侍中（正二品），副职为门下侍郎
- **主要职责**：审议中书省所拟诏敕，如认为不当可予以驳回，确保政策的合理性和可行性

#### 尚书省（执行机构）
- **职能**：负责执行国家政令，统领六部，承担项目落地执行的角色
- **长官**：尚书令（正二品），但唐太宗曾任尚书令，故其后不置，以左右仆射代行其职
- **主要职责**：承宣政令，统领六部九卿等执行机构，是国家行政的核心枢纽

### 2.2 六部架构

#### 吏部（人事管理部门）
- **职能**：负责全国文官的任免、考核、升降、调动等事务
- **长官**：吏部尚书（正三品）
- **下属四司**：吏部司、司封司、司勋司、考功司

#### 户部（资源管理部门）
- **职能**：管理全国土地、户籍、赋税、财政收支等
- **长官**：户部尚书（正三品）
- **下属四司**：户部司、度支司、金部司、仓部司

#### 礼部（文档、规范、用户界面与对外沟通）
- **职能**：主管国家祭祀、礼仪、科举考试及外交事务
- **长官**：礼部尚书（正三品）
- **下属四司**：礼部司、祠部司、膳部司、主客司

#### 兵部（安全和防御部门）
- **职能**：负责武官选用、军队训练、武器装备管理等军务
- **长官**：兵部尚书（正三品）
- **下属四司**：兵部司、职方司、驾部司、库部司

#### 刑部（审计和监督部门）
- **职能**：主管全国司法审判事务，复核刑狱案件
- **长官**：刑部尚书（正三品）
- **下属四司**：刑部司、都官司、比部司、司门司

#### 工部（技术开发和维护部门）
- **职能**：管理国家工程建设、水利、交通、工匠等事务
- **长官**：工部尚书（正三品）
- **下属四司**：工部司、屯田司、虞部司、水部司

## 3. 运行机制

### 3.1 决策流程
1. **中书出令**：中书省根据皇帝旨意或国家需要，起草诏令
2. **门下审令**：门下省对诏令进行审核，如有异议可封驳
3. **尚书行令**：经门下省审核通过后，交由尚书省执行

### 3.2 相互制约关系
- 中书省负责决策，但其决策需经门下省审核
- 门下省有封驳权，但不能自行决策
- 尚书省负责执行，但无决策权
- 三省相互配合又相互制约，有效防止个人专权

### 3.3 任务生命周期
```
任务创建 → 中书决策 → 门下审议 → 尚书分配 → 六部执行 → 结果验收
```

### 3.4 状态流转机制
- **Pending**: 待处理状态
- **Taizi**: 太子监国状态（代理处理）
- **Zhongshu**: 中书省处理状态
- **Menxia**: 门下省审议状态
- **Assigned**: 已分配给尚书省状态
- **Review**: 复核状态
- **Done**: 完成状态

### 4.2 任务ID生成机制
- 使用 `generate_task_id.py` 脚本
- 格式：`JJC-YYYYMMDD-HHMMSSmmm`
- 确保全局唯一性和时间顺序性
- 通过原子操作防止并发冲突

## 5. 技术实现

### 5.1 核心流程
通过 `dispatch_pending_agents.py` 实现：
- 监控任务状态变化
- 根据状态自动分发给对应代理
- 实现催办机制防止任务停滞
- 检测代理活跃度避免重复派发

### 5.2 文件锁定机制
- 使用 `file_lock.py` 实现原子JSON读写
- 防止并发访问导致的数据损坏
- 确保状态更新的一致性

### 5.3 任务状态持久化
- 任务数据存储于 `data/tasks_source.json`
- 通过原子更新保证数据完整性
- 支持任务流转日志记录

### 5.4 通信协议
- 标准化的消息格式
- 包含任务ID、状态、流转记录等关键信息
- 支持各部门间的有效协作

## 6. 工作协议

### 6.1 基本原则
1. 接到任务先回复"已接旨"。
2. 输出必须包含：任务ID、结果、证据/文件路径、阻塞项。
3. 需要协作时，回复尚书省请求转派，不跨部直连。
4. 涉及删除/外发动作必须明确标注并等待批准。

### 6.2 看板操作
所有看板操作必须用 `kanban_update.py` CLI 命令：

#### 接任务时
```bash
python3 scripts/kanban_update.py state JJC-xxx Doing "礼部开始执行[子任务]"
python3 scripts/kanban_update.py flow JJC-xxx "礼部" "礼部" "▶️ 开始执行：[子任务内容]"
```

#### 完成任务时
```bash
python3 scripts/kanban_update.py flow JJC-xxx "礼部" "尚书省" "✅ 完成：[产出摘要]"
```

### 6.3 实时进展上报
执行任务过程中，必须在每个关键步骤调用 `progress` 命令上报当前思考和进展：

```bash
# 开始撰写
python3 scripts/kanban_update.py progress JJC-xxx "正在分析文档结构需求，确定大纲" "需求分析🔄|大纲设计|内容撰写|排版美化|提交成果"

# 撰写中
python3 scripts/kanban_update.py progress JJC-xxx "大纲确定，正在撰写核心章节" "需求分析✅|大纲设计✅|内容撰写🔄|排版美化|提交成果"
```

## 9. 历史意义与价值

三省六部制是中国古代政治制度的重大创新，实现了决策、审议、执行的相对分离，提高了行政效率，加强了中央集权，在中国政治制度史上具有重要地位，对后世影响深远。本项目成功地将传统官僚体系的组织智慧应用于现代AI代理协作中，形成了一个结构清晰、流程规范、自动化的任务管理系统。
